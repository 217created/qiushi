# 求是前端重构 — 产品需求文档

## Problem Statement

求是 (QiuShi) 的后端提供了以哲学思辨为框架的 AI 思考能力：概念炼金术、历史透视、矛盾追踪、跨会话记忆、多哲学流派知识库。但当前前端只是最简陋的 Q&A 聊天界面——三段式输出被扁平渲染为普通文字，思考过程对用户完全不可见，交互模式仅停留在"发消息→收回复"。用户的使用感受与 ChatGPT/DeepSeek 等通用对话产品没有任何区别，后端的大量差异化能力完全没有穿透到用户体验层。

产品核心矛盾是：**后端像一个哲学思考引擎，前端却套了一个通用 AI 聊天的壳。** 用户感觉不到"哲学思辨"这个核心定位，产品没有独特的体验记忆点。

## Solution

围绕"哲学思辨"重新设计前端体验。核心原则：**让思考过程可见，让交互超越对话。**

具体来说：

1. **思考可视化** — 把 AI 的思考步骤（概念松动 → 矛盾分析 → 知识调用 → 自我反驳 → 综合判断）逐层展示给用户，让用户看到 AI 是如何处理他的问题的，而不仅仅是看到最终答案
2. **模式多样化** — 提供三种核心交互模式，覆盖用户从"迷茫"到"想清楚"的完整路径
3. **视觉调性差异化** — 禅宗庭院 + 哲学书房的气质：极致的克制和留白，文字自身的力量感，深沉的暖色调，优雅的动效，完全不做 AI chatbot 通用的毛玻璃/霓虹/未来感设计
4. **思维画像前端化** — 把后端已有的矛盾追踪、决策执行率等能力做成可浏览、可交互的界面

## 用户优先级

V1 用户分层，按优先级排序：

1. **哲学与思辨爱好者**（核心种子用户） — 对存在主义、辩证法、斯多葛等哲学流派有天然兴趣，追求深度思考和自我认知。他们会被产品的思想深度和哲学气质吸引
2. **经历迷茫/决策困难的普通人**（扩展用户） — 面临职业选择、人生困惑、焦虑等实际问题，寻求的不只是答案，而是帮自己想清楚的工具。他们会被产品的实用性和思维框架价值留住

产品定位既要满足核心用户的哲学调性追求（不能"俗化"），也要让扩展用户能无门槛地解决实际问题（不能"太学术"）。

## User Stories

1. 作为一个对哲学思辨感兴趣的用户，我想在打开页面时感受到这个产品的独特气质，以便知道它不是又一个 AI chatbot
2. 作为一个面临决策困难的用户，我想输入我的困惑并看到 AI 有条理地帮我分析，以便获得实际的决策参考
2. 作为一个正在做决策的用户，我想输入我的困惑并看到 AI 的分析过程（概念理解→矛盾识别→历史透视），以便理解 AI 是怎么思考我的问题的
3. 作为一个寻求自我认知的用户，我想看到 AI 回复中"分析""反思""总结"三段结构清晰呈现，以便快速把握思考脉络
4. 作为一个有深度思考需求的用户，我想看到 AI 引用的哲学知识和匹配的比喻，以便获得超出直觉的洞见
5. 作为一个长期用户，我想查看我的"思维画像"（矛盾类型演变、决策执行率、对话统计），以便看到自己思维模式的变化
6. 作为一个想深入探索问题的用户，我想启动"苏格拉底追问"模式，以便让 AI 通过多轮追问帮我厘清核心矛盾
7. 作为一个想获得多元视角的用户，我想启动"多学派辩论"模式，以便从斯多葛、辩证唯物、存在主义等不同哲学立场看同一个问题
8. 作为一个喜欢阅读和思考的用户，我想浏览思辨卡片库（自由、选择、意义等概念），以便获取哲学智慧
9. 作为一个需要隐私感的用户，我想所有数据只存在于我的本地设备上，以便放心倾诉真实的困惑
10. 作为一个有配置需求的用户，我希望能在前端页面配置 API Provider、模型、Obsidian Vault 等设置，以便不用切到命令行
11. 作为一个连续使用的用户，我想在多会话之间切换并看到每个会话的清晰标题和历史，以便管理不同话题的思考
12. 作为一个初次体验的用户，我想看到功能引导，以便快速理解这个产品能做什么、和通用 AI 有什么不同

## Implementation Decisions

### 1. 交互模式系统

前端引入"模式"概念，取代当前单一的 chat 模式。每个模式是一个独立的路由/视图，共享底层会话和消息基础设施。

| 模式 | 核心交互 | 优先级 |
|------|---------|--------|
| 思辨对话 | Q&A + 三段式可视化 + 思考过程展示 | P0 |
| 苏格拉底追问 | AI 逐步追问用户，多轮深入 | P1 |
| 多学派辩论 | 同一问题多哲学流派并行分析 | P2 |
| 思辨卡片 | 概念浏览和展开阅读 | P3 |
| 思维画像 | 个人思维数据可视化仪表盘 | P0 |

### 2. 思考过程可视化（核心差异化）

当前 AI 回复只展示三段式文本。重构后，在 AI 消息内部展示"思考过程面板"——一个可折叠/可展开的区域，显示：

- 概念炼金术：对用户用词的松动和改写（"压力"这个词是否准确？→ 可能是焦虑、过载、迷惘）
- 矛盾检测：检测到的矛盾类型（如果有历史数据）
- 知识引用：匹配到的哲学智慧来源
- 注入比喻：使用了什么比喻来帮助理解

展开状态默认展示简化的卡片/标签，点击可展开详细内容。用 2-3 个词/短语的摘要帮助用户扫读，而不是大段文字。

思考过程面板的设计原则：
- 默认展示紧凑摘要（3-5 个标签/徽标）
- 用户可展开查看详细内容
- 仅在有内容时才出现（没有矛盾就不展示矛盾模块）
- 用视觉设计区分"思考过程"和"回答内容"

### 3. AI 消息三段式增强

当前三段式已经在渲染，但呈现平淡。重构后：

- **分析段**：正文排版，段首有标记色点 + 标签
- **反思段**：可折叠，默认收起，视觉上用斜体 + 灰标区分。另提供"展开全部"选项
- **总结段**：用引用块/框线框起来，视觉上有"结论"的心理暗示
- 三段之间用微妙的分隔线区分，深度选择器调整为更明显的视觉控件

思考过程面板嵌入在 AI 消息的顶部或侧边，用户优先看到"它是怎么想的"，再看到"它得出的结论"。

### 4. 主题系统

保留现有 CSS 变量体系（stone + amber 色调），但在视觉传达上增加：

- 字体的质感和层级：标题用毛体大字做品牌锚点，正文用系统字体的最佳字距/行距
- 更克制的色彩运用：除了突出重点信息（标签、引用、强调）外，大面积使用中性色
- 动画的系统化：不是用 GSAP 等重型库，而是用 CSS transition + animation 体系，控制在几个关键点（消息出现、面板展开、模式切换）
- 间距的层次化：严格使用 4/8/12/16/24/32/48/64 间距体系

### 5. 路由系统

从当前单页无路由，升级为：

```
/            → 首页（思辨对话模式）
/profile     → 思维画像
/cards       → 思辨卡片库
/cards/:id   → 单张卡片详情
/settings    → 设置页
```

使用 React Router 或简单的 hash-based 路由。不引入重型路由方案。

### 6. 技术栈

继承现有的 Vite + React 19 + TypeScript + Tailwind v4 + Phosphor Icons。

新增依赖：
- `react-router`（或 hash-routing 自定义方案）
- 不引入任何 UI 组件库、动画库、状态管理库

状态管理继续使用 `useReducer` + Context，每个页面/模式有自己的 reducer。

### 8. API 对接层扩展

当前只对接了 `POST /api/chat`（SSE）和 `GET /api/status`。需要新增：

| 接口 | 用途 | 状态 |
|------|------|------|
| `POST /api/chat` | 思辨对话（SSE 流式） | 已有 |
| `POST /api/chat/sync` | 同步对话 | 已有 |
| `GET /api/profile` | 思维画像 | 已有 |
| `GET /api/status` | 健康检查 | 已有 |
| `GET /api/knowledge` | 知识检索 | 已有 |
| `POST /api/chat/dialectic` | 苏格拉底追问链 | 待扩展 |
| `POST /api/chat/council` | 多学派辩论 | 待扩展 |
| `GET /api/card/:concept` | 思辨卡片 | 待扩展 |
| `GET /api/config` | 获取配置 | 待扩展 |
| `PUT /api/config` | 更新配置 | 待扩展 |

与后端协商协议格式，保持与现有 SSE 模式一致。

### 9. 数据层

保留 localStorage 持久化方案。每个模式的消息数据可独立序列化。

不需要后端数据库，所有用户数据存本地。

## Testing Decisions

1. **测试原则**：仅测试组件的外部行为（渲染输出 + 用户交互），不测试实现细节（state 结构、reducer 分支）
2. **测试工具**：Vitest + React Testing Library
3. **测试范围**：
   - AiMessage（含 ThinkingPanel 展开/折叠行为）
   - ModeSelector（模式切换正确）
   - ChatInput（发送/禁用/快捷键）
   - DialecticFlow（轮次流转正确）
4. **不覆盖**：
   - SSE 流式 hook（依赖真实网络，测试成本 > 收益，通过集成测试覆盖）
   - 整体的路由集成测试

## Out of Scope

- PWA / 离线支持
- 国际化（i18n）
- Electron 桌面应用
- 移动 App
- 服务端渲染（SSR）/ Next.js
- 后端逻辑修改（引擎、LLM 调用、知识库等）——除非前端需要新接口
- 用户系统 / 多用户 / 账户体系

## Further Notes

- 本 PRD 聚焦前端体验重构。后端的引擎能力（概念炼金术、矛盾检测、历史透视、比喻注入等）已经成熟，不需要修改
- 视觉上参考方向是"禅宗庭院 + 哲学书房"气质，但具体 Design Token 需要在后续设计阶段确定
- V1 重构优先做 P0 内容：思辨对话的思考可视化 + 思维画像升级。P1-P2 在 V1 稳定后逐步落地
- 苏格拉底追问链和多学派辩论需要后端暴露对应的 API 端点（当前只在 CLI 中支持），需要与后端协商接口设计
