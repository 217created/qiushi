# 我如何把一个过度设计的 AI 项目砍掉一半代码，反而更好用了

## 最初的问题

我写了一个 AI 思辨助手叫"求是"。一开始它有一个复杂的**两段式流水线**：先让 LLM 做分析（不限格式），再把分析结果发给同一个 LLM 做"口语化重写"。

为什么这么做？因为最开始我用的 **qwen3:8b 模型**（只有 80 亿参数），它管不住自己的输出格式。即使 system prompt 里写了三遍"不要列大纲"，它还是给你列"一、二、三、综上所述"。所以我用两段式来绕过这个限制——第一段让你列，第二段再让你说人话。

两段式确实解决了问题，但它带来了新的问题：

- **每次对话 = 两次 LLM 调用 = 2 倍延迟 + 2 倍成本**
- **第二阶段是有损压缩**——分析的细微洞察可能被重写丢失
- **代码复杂化**——维护两套 system prompt，两个温度参数

## 转折点

我升级到了 DeepSeek，一个指令遵循能力极强的模型。

然后我问了自己一个尴尬的问题：**我当初做两段式的理由——模型控不住格式——是不是已经不成立了？**

答案是：是的。

我做了 A/B 测试：同样的输入，一段式 vs 两段式，对比输出质量。一段式（直接让 DeepSeek 以"求是"人格输出）不仅不比两段式差，在某些情况下还更好——因为它保留了分析的原始质感，没有被"重写"环节磨平。

于是我砍掉了整个第二阶段。

## 不只是砍，是重构

砍掉两段式只是开始。我又检查了整个项目，发现类似的问题还有很多：

**StageManager 状态机** —— 4 阶段状态机（缺事实→事实充分→有方案→执行中）听起来很聪明，但现实中的对话不是线性的。用户今天在阶段 2，下周回来又在阶段 0（新问题）。状态转移用关键词触发——用户说"我做了顿饭"触发"做了"关键词，从阶段 0 跳到阶段 1。这是假阳性。我删除了整个 StageManager 文件，把主动对换成了系统的分阶段引导写的直接提示词。

**profile.py + obsidian.py** —— 两个文件加起来不到 200 行代码，完全可以合并到 engine.py，减少文件数量，降低理解负担。

**scene_affinity 硬编码** —— 我写了一个功能：检测用户输入的关键词（"职业"→优先匹配毛选，"人生"→优先匹配存在主义）。这个功能的问题在于它是**先验的**——你凭什么认为职业问题就应该匹配毛选？最好的方法是删除这个硬编码，在 system prompt 里直接告诉 LLM："你脑子里有多种思想资源，选最合适的。"

**严厉指数自动检测** —— 用关键词检测用户是否执行了建议。用户说"我试了但失败了"→触发"做了"→严厉指数-1。这个检测 75% 的时间都在报假。删除。保留用户画像只做对话记录。

## 结果

- **核心代码从 9 个 Python 文件缩减到 4 个**
- **API 成本降低一半**（一次调用 vs 两次调用）
- **响应延迟减半**
- **代码更容易理解和调试**

## 更重要的教训

我学到的东西不止是关于代码的：

1. **为特定模型设计的架构是有保质期的。** 你的模型升级了，架构可能也应该升级。
2. **两段式架构是一个"拐杖"，不是一条"腿"。** 拐杖可以在你需要的时候帮你走路，但你不能永远拄着。需要定期检查：你现在还需要它吗？
3. **砍功能比加功能难，但也更有价值。** 我花了更多时间决定删除什么，而不是决定添加什么。
4. **"先验的智能"往往不智能。** 那些"看起来聪明"的设计——状态机、权重矩阵、规则引擎——很多时候不如一个简单直接的 prompt 有效。让 LLM 自己判断，而不是替它判断。

## 现在的求是

现在的 "求是" 是一个**单段式**的思辨引擎：

- 用户输入 → 知识检索 → 语境比喻注入 → 强制三段式分析（分析/反驳/总结）→ 代码解析展示
- 可拔插 LLM（DeepSeek / OpenAI / Claude / Ollama / LiteLLM 100+ 模型）
- 中文双字分词的关键词检索（对 33 篇哲学知识库足够精确）
- 概念炼金术（代码检测抽象词并强制重新定义）
- 强制自我反驳（每次分析后 LLM 必须推翻自己一次）
